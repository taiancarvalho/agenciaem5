# Agency OS — Spec

# Vision

## O que é

Agency OS é um sistema operacional para agências de marketing digital. Não é só um conjunto de prompts nem apenas um gerador de campanhas. A proposta é criar uma máquina operacional que una estratégia, operação, documentação, playbooks, automação e agentes especializados em um único ecossistema.

## Objetivo principal

Transformar a agência em um sistema estruturado, replicável, escalável e treinável.

## O problema que resolve

A maioria das agências opera com conhecimento solto, processos frágeis, dependência excessiva de pessoas específicas, pouca documentação, execução inconsistente e dificuldade para escalar qualidade. O Agency OS nasce para reduzir caos operacional, padronizar execução, melhorar a própria agência e depois melhorar a operação dos clientes.

## Princípio central

O Agency OS não deve nascer como um sistema para rodar campanhas. Ele deve nascer como um sistema operacional completo da agência.

---

# Fundamentos conceituais

## O sistema não é sobre agentes isolados

O maior aprendizado extraído do AIOX é que valor não está em ter muitos agentes inteligentes, mas em ter um sistema com:

- papéis claros
- comandos definidos
- handoffs explícitos
- artefatos padronizados
- limites rígidos de escopo
- tarefas especializadas desacopladas do agente-base

## Fórmula estrutural do agente

Cada agente do sistema deve seguir a lógica:

1. Identidade
2. Quando usar
3. Quando não usar
4. Princípios de decisão
5. Comandos
6. Tasks e dependências
7. Templates
8. Ferramentas
9. Regras de handoff
10. Critérios de saída

## Princípio de contexto

Planejamento pode ser amplo. Execução precisa ser estreita. O executor não deve carregar contexto desnecessário.

## Regra operacional

Conversa = exploração.

Notion = verdade oficial.

# Comparativo estrutural com o AIOX

## Leitura estratégica

O AIOX foi analisado não para ser copiado, mas para servir como referência estrutural. A conclusão principal é que o AIOX não é um monte de prompts. Ele é um sistema composto por papéis, comandos, handoffs, artefatos, tasks, templates, workflows e limites de atuação.

## O que foi analisado

Foram analisados três perfis do AIOX:

- aiox-master
- pm
- dev

Esses três perfis representam três camadas fundamentais do sistema.

## Camadas identificadas

### Camada 1 — Orquestração

Inspirada no aiox-master.

Responsável por:

- governança do sistema
- criação e modificação de componentes
- execução e validação de workflows
- registro de capacidades
- decisão de sequência
- orquestração entre agentes

### Camada 2 — Definição

Inspirada principalmente no PM.

Responsável por:

- entendimento do problema
- definição de direção
- estruturação do escopo
- priorização
- métricas
- planejamento
- transformação de briefing em plano

### Camada 3 — Execução

Inspirada no DEV.

Responsável por:

- operação prática
- execução de subtarefas
- verificação
- correção
- retomada
- entrega

## Comparativo resumido

### AIOX Master

Papel: orquestrador e arquiteto do sistema.

Não é um PM mais forte. É um meta-agente de framework.

Tem capacidade de criar, modificar, validar e governar o sistema.

### PM

Papel: definidor de direção.

Transforma ideia em especificação, estrutura escopo e organiza prioridades.

### DEV

Papel: executor operacional.

Pega uma tarefa definida, executa passo a passo, testa, corrige e entrega.

## Insight principal do comparativo

O AIOX não é sobre senioridade. Não é master mais sênior que dev. São tipos diferentes de máquina:

- master = governança e orquestração
- pm = definição e estruturação
- dev = execução controlada

Esse mesmo modelo deve orientar o Agency OS.

# Arquitetura do Agency OS

## Camada 1 — Orquestração

### Agency Master

Função:

- receber demandas
- classificar tipo de fluxo
- decidir quais agentes entram
- controlar handoffs
- criar ou modificar componentes do sistema
- manter coerência da arquitetura
- manter registry de capacidades
- governar workflows ponta a ponta

### Tipo de comandos esperados

- create-agent
- modify-agent
- create-workflow
- validate-workflow
- run-workflow
- list-capabilities
- register-playbook
- impact-analysis
- orchestrate-client-flow

## Camada 2 — Definição

### Estrategista de Agência

Função:

- posicionamento
- definição de nicho
- ICP
- proposta de valor
- diferenciação
- estratégia comercial
- lógica de oferta
- diagnóstico e direcionamento

### Atendimento / Briefing Manager

Função:

- coletar briefing
- organizar contexto
- detectar lacunas
- preparar insumos para estratégia

### Sales Optimizer

Função:

- melhorar script comercial
- analisar reuniões e calls
- extrair objeções e gargalos
- melhorar oferta, pitch e narrativa de venda

## Camada 3 — Execução

### Gestor de Tráfego

Função:

- executar operação de mídia
- subir campanhas
- auditar contas
- validar tracking
- gerar relatórios
- otimizar

### Copywriter

Função:

- criar ângulos
- escrever anúncios
- estruturar ofertas
- quebrar objeções
- produzir copies de LP e funil

### Designer / Creative Director

Função:

- transformar briefing em direção visual
- estruturar criativos
- validar coerência visual
- produzir variações

### QA de Campanha

Função:

- revisar entregáveis
- verificar congruência entre estratégia, copy, criativo e mídia
- separar erro crítico de melhoria opcional
- validar qualidade antes da publicação

### Ops / System Builder

Função:

- estruturar pastas
- organizar workspace
- padronizar arquivos
- preparar base operacional do sistema

# Estrutura de componentes do sistema

## O que não deve ficar só no agente-base

Instruções muito específicas por canal, ferramenta ou contexto não devem ficar todas enfiadas no prompt principal do agente. Elas devem ser distribuídas em componentes auxiliares.

## Princípio de modularidade

O sistema deve ser capaz de suportar apenas os canais e rotinas realmente usados pela agência. Exemplo: se a agência não opera Pinterest, isso não precisa existir na configuração ativa.

# Flows do Agency OS

## Flow 0 — Agency Setup

### Definição

Fluxo inicial de setup da própria agência. Antes de atender clientes, a própria agência deve ser tratada como seu primeiro produto.

### Objetivo

Estruturar a agência como uma máquina operacional clara, posicionada e organizada.

### Escopo inicial

- definição do nome da agência
- definição de nicho ou operação genérica
- definição de posicionamento
- definição de comunicação
- branding
- estratégia comercial
- oferta
- script de vendas
- organização do workspace da agência
- estrutura de arquivos e pastas
- aprendizado contínuo a partir de reuniões gravadas e transcritas

### Observação estratégica

Este fluxo existe para melhorar a própria agência antes da operação com clientes.

### Sub-blocos identificados

#### Posicionamento

- nicho versus genérico
- ICP
- proposta de valor
- diferenciação

#### Branding

- nome
- comunicação
- tom de voz
- identidade

#### Estratégia comercial

- oferta
- script de venda
- apresentação
- melhoria contínua via análise de calls

#### Estrutura interna

O sistema precisa criar organização real de documentos e arquivos, para que o usuário possa localizar relatórios, briefings, criativos e materiais por pasta de forma prática.

#### Sistema de aprendizado da agência

Loop proposto:

reunião → transcrição → análise → melhoria → novo script

### Agentes envolvidos

- Agency Master
- Estrategista de Agência
- Sales Optimizer
- Ops / System Builder

### Saídas esperadas

- posicionamento definido
- oferta definida
- narrativa comercial melhorada
- estrutura interna organizada
- pasta base da agência preparada
- materiais comerciais evolutivos

# Estrutura sugerida de workspace

```
/workspace/
  /admin/
  /clientes/
    /cliente-x/
      /briefing/
      /relatorios/
      /criativos/
      /campanhas/
  /templates/
  /playbooks/
```

## Intenção

Essa estrutura existe para que qualquer documento gerado automaticamente tenha lugar previsível. Exemplo: se um relatório foi gerado para um cliente, ele deve estar na pasta do cliente, em relatórios, com padrão de data e nome previsível.

# Regras de desenho do sistema

## Regra 1

Não copiar o AIOX literalmente. Extrair padrão estrutural.

## Regra 2

Não construir agentes apenas com base em cargos humanos. Em alguns casos, o papel é melhor definido por tipo de trabalho do que por profissão formal.

## Regra 3

Não misturar orquestração, definição e execução no mesmo agente sem necessidade.

## Regra 4

Instruções de canal, colunas, templates, integrações e lógicas operacionais específicas devem ficar distribuídas em tasks, workflows, data e templates.

## Regra 5

O executor deve trabalhar com contexto reduzido.

## Regra 6

A agência deve ser tratada como o primeiro cliente do próprio sistema.
